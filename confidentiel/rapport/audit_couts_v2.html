Techniques d'optimisation déployées

Batch scoring — 1 appel pour N biens

Nouvelle fonction scorer_biens_batch_claude() qui envoie tous les biens d'un prospect en un seul appel. 605 appels → 121 par run-all. Même barème, mêmes règles. Fallback automatique sur appels individuels si le JSON batch est invalide.

Prompt caching Anthropic

Les 2 031 tokens statiques (barème, règles, anti-biais) marqués cache_control: ephemeral. Facturés $3.75/M à la création, puis $0.30/M en lecture (10% du prix normal). Économie de 81% sur la partie statique du prompt.

Helpers extraits et centralisés

_build_focus_destination(), _build_section_contraintes(), _parse_claude_json(), _track_usage() — partagés entre l'appel batch et le fallback individuel. Zéro duplication de logique.

Fallback automatique préservé

Si le batch échoue (JSON invalide, count mismatch), scorer_biens() retombe silencieusement sur les appels individuels parallèles (ThreadPoolExecutor max 4). Aucun risque de perte d'analyse.

Limites et points de vigilance

Point Impact Mitigation
Cache TTL : 5 minutes Au-delà de 5 min d'inactivité, le cache expire → prochain appel recréé le cache ($3.75/M) Léger surcoût ponctuels. Économie nette reste significative car run-all est continu (~22s/prospect).
Output plus long en batch 5 biens × ~250 tokens output = 1 250 tokens vs 5 × 180 = 900 tokens avant Surcoût output de ~$0.005/prospect, compensé largement par la réduction des input.
JSON batch potentiellement invalide Si Claude hallucine le format tableau → fallback individuel déclenché Test sur 5 biens : 0 erreur. Fallback automatique sans interruption de service.
max_tokens = 4 096 Pour max_biens=8 biens : 500×8+300 = 4 300 tokens → plafonné à 4 096 max_biens est à 5, donc 2 800 tokens max output — aucun risque de troncature.
